# Release 9.x.x

Version 9.x.x of the Data Space Connector contains a number of breaking changes and improvements. They will be listed below, together with required actions for migration.

## Decentralized-IAM Integration

All IAM and authorization components are now consolidated under a new [`decentralized-iam`](https://github.com/FIWARE/decentralized-iam) umbrella chart. The following top-level chart dependencies have been removed and are now managed under `decentralizedIam`:

| Removed (8.5.2) | New location (9.0.0) |
|---|---|
| `mysql` | N/A — replaced by postgres-operator |
| `postgresql` (for auth/PAP/Keycloak/IdentityHub) | N/A — replaced by postgres-operator |
| `postgis` (for Scorpio) | N/A — replaced by postgres-operator |
| `vcverifier` | `decentralizedIam.vcAuthentication.vcverifier` |
| `credentials-config-service` | `decentralizedIam.vcAuthentication.credentials-config-service` |
| `trusted-issuers-list` | `decentralizedIam.vcAuthentication.trusted-issuers-list` |
| `dss` | `decentralizedIam.vcAuthentication.dss` |
| `odrl-pap` | `decentralizedIam.odrlAuthorization.odrl-pap` |
| `apisix` (+ OPA sidecar) | `decentralizedIam.odrlAuthorization.apisix` |

:warning: Required action:

Replace all individual IAM component configurations in your values file with the new `decentralizedIam` structure:

```yaml
decentralizedIam:
  enabled: true
  vcAuthentication:
    trusted-issuers-list: ...
    vcverifier: ...
    credentials-config-service: ...
    dss: ...
  odrlAuthorization:
    apisix: ...
    odrl-pap: ...
    opa: ...
    tpp: ...
```

The following top-level values keys are no longer valid and must be removed:

```

```

## Zalando postgres-operator

The chart no longer ships individual Bitnami PostgreSQL or MySQL instances for auth components. Instead, it uses the [Zalando postgres-operator](https://github.com/zalando/postgres-operator) to create and manage a single PostgreSQL cluster with all required databases and users.

The `vc-authentication` subchart (part of `decentralized-iam`) includes `postgres-operator 1.15.1` as a direct dependency. You have two options:

**Option A — Let the chart install the operator (recommended for new installations):**

The postgres-operator chart ships its CRDs in its `crds/` directory, so Helm installs them automatically on `helm install`:

```yaml
decentralizedIam:
  vcAuthentication:
    postgres-operator:
      enabled: true   # default: true
```

> **Helm upgrade caveat:** Helm only installs CRDs on `helm install`, not on `helm upgrade`. If the postgres-operator CRDs change in a future version, apply them manually before upgrading:
> ```bash
> kubectl apply -f https://raw.githubusercontent.com/zalando/postgres-operator/refs/heads/master/charts/postgres-operator/crds/postgresqls.yaml
> kubectl apply -f https://raw.githubusercontent.com/zalando/postgres-operator/refs/heads/master/charts/postgres-operator/crds/operatorconfigurations.yaml
> kubectl apply -f https://raw.githubusercontent.com/zalando/postgres-operator/refs/heads/master/charts/postgres-operator/crds/postgresteams.yaml
> ```

**Option B — Use an externally managed operator:**

```yaml
decentralizedIam:
  vcAuthentication:
    postgres-operator:
      enabled: false
```

The `managedPostgres` section controls the `postgresql.acid.zalan.do` CRD that defines the actual cluster:

```yaml
decentralizedIam:
  vcAuthentication:
    managedPostgres:
      enabled: true
      config:
        teamId: "dsc"   # cluster service will be named: postgres
        numberOfInstances: 1
        postgresql:
          version: "16"
        volume:
          size: 1Gi
        users:
          admin: [superuser, createdb]
          til: [createdb]
          ccs: [createdb]
          pap: [createdb]
          keycloak: [createdb]
          ih: [createdb]
          issuer: [createdb]
        databases:
          tildb: til
          ccsdb: ccs
          papdb: pap
          keycloakdb: keycloak
          ih: ih
          issuer: issuer
```

:warning: Required action:

All database secrets and hostnames have changed. The operator auto-generates one Kubernetes Secret per user following the pattern `{username}.postgres.credentials.postgresql.acid.zalan.do`. Update all secret references:

| Component | Old secret / key | New secret / key |
|---|---|---|
| TIL, CCS | `authentication-database-secret` / `mysql-root-password` | `postgres.postgres.credentials.postgresql.acid.zalan.do` / `password` |
| ODRL-PAP, Keycloak, Scorpio | `database-secret` / `postgres-admin-password` | `postgres.postgres.credentials.postgresql.acid.zalan.do` / `password` |
| IdentityHub | `database-secret` / `postgres-admin-password` | `ih.postgres.credentials.postgresql.acid.zalan.do` / `password` |

Database hostnames and names have also changed:

| Component | Old host / DB | New host / DB |
|---|---|---|
| TIL, CCS | `authentication-mysql` / `tildb`, `ccsdb` | `postgres` / `tildb`, `ccsdb` |
| ODRL-PAP | `postgresql` / `pap` | `postgres` / `papdb` |
| Keycloak | `postgresql` / `keycloak` | `postgres` / `keycloakdb` |
| Scorpio | `data-service-postgis` | `postgres` |
| IdentityHub | *(empty)* | `postgres` |

Note: the ODRL-PAP database was renamed from `pap` to `papdb`. Migrate any existing policy data before upgrading.

## TIL and CCS: MySQL replaced by PostgreSQL

`trusted-issuers-list` and `credentials-config-service` previously used MySQL. They now use PostgreSQL. Liquibase migrations will run against the new databases on first startup.

:warning: Required action:

If you have existing data in the MySQL databases (`tildb`, `ccsdb`), migrate it to the new PostgreSQL instance using a tool such as [pgloader](https://pgloader.io/) before upgrading.

Update the database dialect in your values:

```yaml
decentralizedIam:
  vcAuthentication:
    trusted-issuers-list:
      database:
        dialect: POSTGRES
        host: postgres
        port: 5432
        username: postgres
        existingSecret:
          name: postgres.postgres.credentials.postgresql.acid.zalan.do
          key: password
        name: tildb
```

## Verifier: built-in credentials config server (credentials-config-service no longer required)

The `vcverifier` now ships a built-in config server that reads credential configurations directly from the same PostgreSQL database previously managed by `credentials-config-service`. Deploying `credentials-config-service` as a separate component is therefore no longer necessary.

:warning: Required action:

**1. Configure the verifier's database connection** using the same database that `credentials-config-service` was pointing to (`ccsdb`), and enable the built-in config server:

```yaml
decentralizedIam:
  vcAuthentication:
    vcverifier:
      deployment:
        verifier:
          database:
            persistence: true
            dialect: POSTGRES
            username: postgres
            existingSecret:
              enabled: true
              name: postgres.postgres.credentials.postgresql.acid.zalan.do
              key: password
            host: postgres
            port: 5432
            name: ccsdb
        configServer:
          enabled: true
        configRepo:
          configEndpoint: ""
```

Note that `configRepo.configEndpoint` must be set to an empty string so the verifier uses the built-in config server instead of an external `credentials-config-service` endpoint.

**2. Disable `credentials-config-service`:**

```yaml
decentralizedIam:
  vcAuthentication:
    credentials-config-service:
      enabled: false
```

## Keycloak: keystore password configuration changed

In 8.5.2, the keystore password was injected into Keycloak via the `STORE_PASS` environment variable reading from the `kc-keystore` Kubernetes secret. This has been removed.

In 9.0.0 the keystore credentials are configured directly in values and rendered into the Keycloak realm configuration. Two approaches are supported:

**Option A — `did:elsi` (top-level `elsi` section):**
```yaml
elsi:
  enabled: true
  did: did:elsi:myDid
  keyAlgorithm: EC
  storePath: /path/to/keystore.p12
  storePassword: my-store-password
  keyAlias: my-key-alias
  keyPassword: my-key-password
```

**Option B — generic signing key (`keycloak.signingKey`):**
```yaml
keycloak:
  signingKey:
    storePath: /path/to/keystore.p12
    storePassword: my-store-password
    keyAlias: my-key-alias
    keyPassword: my-key-password
    keyAlgorithm: EC
```

:warning: Required action:

The passwords are rendered inline into the realm ConfigMap. To avoid plain-text passwords in your values file, use `${ENV_VAR}` placeholders and load them from a Kubernetes secret via `keycloak.extraEnvVars`:

```yaml
elsi:
  storePassword: "${STORE_PASS}"
  keyPassword: "${KEY_PASS}"

keycloak:
  extraEnvVars:
    - name: STORE_PASS
      valueFrom:
        secretKeyRef:
          name: my-keystore-secret
          key: store-password
    - name: KEY_PASS
      valueFrom:
        secretKeyRef:
          name: my-keystore-secret
          key: key-password
```

The same pattern applies to Option B using `keycloak.signingKey`.

## Keycloak realm: structured YAML format for realm properties

The following `keycloak.realm` properties now accept native YAML structures in addition to the legacy raw JSON string format. The string format remains fully supported for backwards compatibility.

### `attributes` — dictionary

```yaml
# OLD (string)
keycloak:
  realm:
    attributes: |
    "myKey": "myValue",
    "anotherKey": "anotherValue"'

# NEW (map)
keycloak:
  realm:
    attributes:
      myKey: myValue
      anotherKey: anotherValue
```

### `clientRoles` — dictionary (key → list of role objects)

```yaml
# OLD (string)
keycloak:
  realm:
    clientRoles: |
      "my-client": [{"name": "my-role"}],
      "second-client": [{"name": "my-role-2"}]

# NEW (map)
keycloak:
  realm:
    clientRoles:
      my-client:
        - name: my-role
      second-client:
        - name: my-role-2
```

### `users` — list

```yaml
# OLD (string)
keycloak:
  realm:
    users: |
    {"username": "admin", "enabled": true},
    {"username": "test", "enabled": true}

# NEW (list)
keycloak:
  realm:
    users:
      - username: admin
        enabled: true
      - username: test
        enabled: true
```

### `clients` — list

```yaml
# OLD (string)
keycloak:
  realm:
    clients: '{"clientId": "my-client", "enabled": true}'

# NEW (list)
keycloak:
  realm:
    clients:
      - clientId: my-client
        enabled: true
```

### `clientScopes` — list

```yaml
# OLD (string)
keycloak:
  realm:
    clientScopes: |
    {"name": "my-scope", "protocol": "openid-connect"},
    {"name": "my-scope2", "protocol": "openid-connect"}

# NEW (list)
keycloak:
  realm:
    clientScopes:
      - name: my-scope
        protocol: openid-connect
      - name: my-scope-2
        protocol: openid-connect
```

## Keycloak realm: default values split into extensible properties

Several realm properties that previously shipped hardcoded defaults have been split into a `default*` key (chart-managed defaults) and an `extra*` or override key (user additions). This allows extending the defaults without redefining them entirely.

| Property | Default key | Extension key | Type |
|---|---|---|---|
| Realm roles | `defaultRealmRoles` | `extraRealmRoles` | list |
| Groups | `defaultGroups` | `extraGroups` | list |
| Client scopes | `defaultClientScopes` | `clientScopes` | list |
| Default default client scopes | `defaultDefaultClientScopes` | `extraDefaultClientScopes` | list |
| Default optional client scopes | `defaultOptionalClientScopes` | `extraOptionalClientScopes` | list |
| Client roles | `defaultClientRoles` | `clientRoles` | dict |
| Clients | `defaultClients` | `clients` | list |
| Key providers | `defaultKeyProviders` | `extraKeyProviders` | list |

Example — adding an extra realm role without redefining the defaults:

```yaml
# OLD — had to repeat all existing roles plus the new one
keycloak:
  realm:
    realmRoles: '{"name": "default-roles-provider"}, {"name": "offline_access"}, {"name": "my-custom-role"}'

# NEW — only define what you add
keycloak:
  realm:
    extraRealmRoles:
      - name: my-custom-role
```

Example — adding an extra client scope:

```yaml
keycloak:
  realm:
    clientScopes:
      - name: my-scope
        protocol: openid-connect
        attributes:
          include.in.token.scope: "true"
```

## Keycloak realm: automatic `optionalClientScopes` population for default clients

The chart now automatically populates `optionalClientScopes` for each client defined under `keycloak.realm.defaultClients`. The injected scopes are derived from all three sources:

- `keycloak.realm.defaultClientScopes` — chart-managed default scopes
- `keycloak.realm.clientScopes` — user-defined extra scopes
- Scopes auto-generated from `keycloak.realm.verifiableCredentials`

A scope is only added to `optionalClientScopes` if it is **not already present** in that client's `defaultClientScopes`, avoiding duplicates across both scope lists.

## Keycloak realm: dedicated `verifiableCredentials` property

A new `keycloak.realm.verifiableCredentials` property allows declaring each verifiable credential as a self-contained block. Each entry combines three concerns in one place: the VC realm attributes, the client scope that Keycloak exposes for that credential, and its protocol mappers.

> :bulb: **Recommended over `clientScopes` and `attributes` for VC configuration**
>
> Keycloak versions after 26.3 will introduce a breaking change in how verifiable credentials are configured: VC attributes will no longer be stored as plain realm attributes. Declaring VCs through `verifiableCredentials` ensures the chart can handle that transition transparently. Migrating existing VC client scopes from `clientScopes` to `verifiableCredentials` is recommended — it avoids duplicates and keeps the full VC definition in one place. If you prefer to keep managing the client scope manually under `clientScopes`, set `clientScope.create: false` to prevent the chart from generating it automatically.

Each entry supports the following fields:

| Field | Description |
|---|---|
| `attributes` | Map of VC attributes, flattened as `vc.{id}.{attr}` into the realm attributes. Object values are serialized as JSON strings automatically. |
| `clientScope.create` | Whether to auto-generate a client scope for this VC. Defaults to `true`. Set to `false` to skip creation. |
| `protocolMappers` | List of protocol mappers to attach to the generated client scope. See [oid4vc-protocol-mappers.md](../keycloak/oid4vc-protocol-mappers.md) for available mappers and their configuration. |

```yaml
keycloak:
  realm:
    verifiableCredentials:
      my-credential:
        attributes:
          credential_signing_alg_values_supported: "ES256"
          format: "vc+sd-jwt"
          scope: "MyCredential"
          vct: "MyCredential"
          credential_build_config.signing_algorithm: "ES256"
          credential_build_config.token_jws_type: "vc+sd-jwt"
          credential_build_config.visible_claims: "roles,email"
          # object values are automatically serialized as JSON strings
          credential_build_config.proof_types_supported:
            jwt:
              proof_signing_alg_values_supported:
                - ES256
        clientScope:
          create: true   # default, can be omitted
        protocolMappers:
          - name: email-mapper
            protocol: oid4vc
            protocolMapper: oid4vc-user-attribute-mapper
            config:
              subjectProperty: email
              userAttribute: email
              supportedCredentialTypes: MyCredential
          - name: role-mapper
            protocol: oid4vc
            protocolMapper: oid4vc-target-role-mapper
            config:
              subjectProperty: roles
              clientId: did:web:my-did.example.org
              supportedCredentialTypes: MyCredential
```

The `attributes` block produces the following realm attributes (merged last, taking precedence over `keycloak.realm.attributes`):

```json
{
  "vc.my-credential.credential_signing_alg_values_supported": "ES256",
  "vc.my-credential.format": "vc+sd-jwt",
  "vc.my-credential.scope": "MyCredential",
  "vc.my-credential.vct": "MyCredential",
  "vc.my-credential.credential_build_config.signing_algorithm": "ES256",
  "vc.my-credential.credential_build_config.token_jws_type": "vc+sd-jwt",
  "vc.my-credential.credential_build_config.visible_claims": "roles,email",
  "vc.my-credential.credential_build_config.proof_types_supported": "{\"jwt\":{\"proof_signing_alg_values_supported\":[\"ES256\"]}}"
}
```

When `clientScope.create` is `true` (the default), the chart also generates a client scope entry using `attributes.scope` as the name and attaches the `protocolMappers`. This scope is automatically added to the `optionalClientScopes` of each default client (skipping any already present in that client's `defaultClientScopes`).

:warning: **Migration:** if you previously declared VC-related client scopes under `keycloak.realm.clientScopes`, move them to `verifiableCredentials` and remove the duplicate entries from `clientScopes` to avoid the scope appearing twice in the realm.

## did-helper: alias changed and new certificate serving capability

The `did-helper` chart alias and condition have changed:

| | 8.5.2 | 9.0.0 |
|---|---|---|
| Alias | `did-helper` | `did` |
| Condition | `did-helper.enabled` | `did.enabled` |

`did-helper` is a lightweight server that generates and serves a DID document at `/.well-known/did.json`. It supports two modes for key material:

- **`generateKey` (default):** Generates a new EC (or RSA) key pair and self-signed certificate on startup. Useful for development and testing.
- **`provideKeystore`:** Uses an existing PKCS12 keystore provided via a Kubernetes secret. Useful for production.

In 9.0.0, the helper also exposes `GET /.well-known/tls.crt` and can serve a certificate issued by an external CA (e.g. cert-manager) by mounting the TLS secret as a volume:

```yaml
did:
  enabled: true
  ingress:
    enabled: true
    className: nginx
    annotations:
      cert-manager.io/issuer: selfsigned-issuer
      cert-manager.io/common-name: did-provider.127.0.0.1.nip.io
      cert-manager.io/private-key-algorithm: ECDSA
    hosts:
      - host: did-provider.127.0.0.1.nip.io
        paths:
          - path: /
            pathType: ImplementationSpecific
    tls:
      - secretName: did-provider.127.0.0.1.nip.io-tls
        hosts:
          - did-provider.127.0.0.1.nip.io
  volumes:
    - name: certs
      secret:
        secretName: did-provider.127.0.0.1.nip.io-tls
        items:
          - key: tls.crt
            path: tls.crt
  volumeMounts:
    - name: certs
      mountPath: /certs
  config:
    server:
      hostUrl: "https://did-provider.127.0.0.1.nip.io"
      certPath: "/certs/tls.crt"
    generateKey:
      enabled: false
```

:warning: Required action:

Rename `did-helper` to `did` in your values file. The top-level `didJson` section has also been removed — replace it with the `did` subchart as shown above.

## cert-manager integration

9.x.x adds optional [cert-manager](https://cert-manager.io) support directly as a chart dependency. It can manage TLS certificates for all connector ingresses and replace the previous approach of manually generating and distributing certificates via the `helpers/certs` scripts (now removed).

Two new values sections are available:

**`cert-manager`** — installs cert-manager itself as a subchart (disabled by default):

```yaml
cert-manager:
  enabled: true
  crds:
    enabled: true   # installs cert-manager CRDs
  startupapicheck:
    enabled: false
```

**`certManagerResources`** — deploys ClusterIssuers after cert-manager is ready (disabled by default):

```yaml
certManagerResources:
  enabled: true
  namespace: cert-manager   # namespace where cert-manager is deployed
  type: selfsigned          # "selfsigned" or "prod"
  acmeEmail: admins@example.com   # required for "prod" only
  ingressClass: nginx             # required for "prod" only
```

Two issuer types are supported:

- **`selfsigned`** — creates a self-signed CA and a `selfsigned-issuer` ClusterIssuer. Suitable for development and internal environments. Does not require external DNS or HTTP challenge resolution.

- **`prod`** — creates a Let's Encrypt production `prod-issuer` ClusterIssuer using HTTP-01 ACME challenges. Requires a publicly reachable ingress and a valid domain.

Once a ClusterIssuer is available, annotate your ingresses to request certificates automatically. The `selfsigned-issuer` is already used by `did-helper` in the example shown in the `did-helper` section above:

```yaml
ingress:
  annotations:
    cert-manager.io/issuer: selfsigned-issuer        # for selfsigned
    # cert-manager.io/cluster-issuer: prod-issuer    # for Let's Encrypt
```

> **Note:** `certManagerResources` uses Helm post-install/post-upgrade hooks to deploy the ClusterIssuers after cert-manager is ready. If you enable both `cert-manager` and `certManagerResources` in the same install, the issuers will be created automatically in the correct order.

## Grafana Tempo Integration (9.2.0)

9.2.0 adds optional in-cluster **Grafana Tempo** trace storage and **Grafana** dashboard support, building on the OpenTelemetry tracing infrastructure introduced in 9.1.0. The integration is fully opt-in — no existing values are modified or removed.

### New optional subchart dependencies

| Subchart | Chart | Version | Repository |
|---|---|---|---|
| `tempo` | `grafana/tempo` | `1.24.4` | `https://grafana.github.io/helm-charts` |
| `grafana` | `grafana/grafana` | `10.5.15` | `https://grafana.github.io/helm-charts` |

Both are disabled by default and gated by `tempo.enabled` / `grafana.enabled`.

### Auto-wired Collector → Tempo pipeline

When `tracing.enabled=true` and `tempo.enabled=true`, the chart renders a custom OTEL Collector ConfigMap (`otel-collector-config-cm.yaml`) that automatically injects an `otlp/tempo` exporter targeting the in-cluster Tempo gRPC endpoint. No manual Collector configuration is required — the existing `opentelemetry-collector.config` values are used as a base and the Tempo exporter is appended at render time.

When `tempo.enabled=false`, the ConfigMap renders the Collector configuration as-is (identical behaviour to 9.1.0).

### Auto-provisioned Grafana datasource

When both `grafana.enabled=true` and `tempo.enabled=true`, the chart renders a datasource ConfigMap (`grafana-tempo-datasource-cm.yaml`) labelled `grafana_datasource: "1"`. The Grafana sidecar picks it up automatically and registers Tempo as the default datasource. Operators can browse traces in the Grafana Explore view immediately after deployment.

### Enabling the full stack

```yaml
tracing:
  enabled: true
tempo:
  enabled: true
grafana:
  enabled: true
```

This deploys: workloads with OTEL instrumentation → OTEL Collector → Grafana Tempo → Grafana dashboard. No additional configuration is needed for a working development/test setup.

### New values keys

| Key | Type | Default | Description |
|---|---|---|---|
| `tempo.enabled` | bool | `false` | Toggle the bundled Grafana Tempo subchart |
| `tempo.tempo.retention` | string | `"48h"` | Trace retention period |
| `tempo.tempo.receivers.otlp.protocols.grpc.endpoint` | string | `"0.0.0.0:4317"` | OTLP gRPC listen address |

## OpenTelemetry Distributed Tracing (9.1.0)

9.1.0 adds optional end-to-end distributed tracing across the FIWARE Data Space Connector using [OpenTelemetry](https://opentelemetry.io/). All first-party workloads (IdentityHub, fdsc-edc) and third-party subcharts that support native OTEL configuration (Keycloak, Scorpio, tm-forum-api, contract-management, marketplace) can export traces through the OTLP protocol to any compatible backend (Jaeger, Grafana Tempo, Honeycomb, etc.).

An optional [OpenTelemetry Collector](https://opentelemetry.io/docs/collector/) subchart is included for in-cluster trace aggregation and routing.

### Enabling tracing

Set `tracing.enabled: true` to inject `OTEL_*` environment variables into all supported workloads and enable the OpenTelemetry Java agent for IdentityHub:

```yaml
tracing:
  enabled: true
  exporter:
    otlp:
      endpoint: ""          # defaults to the bundled collector
      protocol: "grpc"      # or "http/protobuf"
      insecure: true
  sampler: "parentbased_traceidratio"
  samplerArg: "1.0"
  propagators: "tracecontext,baggage"
  resourceAttributes: {}

opentelemetry-collector:
  enabled: true
```

### Per-component tracing

Each component can override the global tracing toggle independently via its own `tracing.enabled` key:

```yaml
identityhub:
  tracing:
    enabled: true           # override global toggle
    serviceName: "my-identityhub"
    javaagent:
      enabled: true         # attach the OTEL Java agent via init container

fdsc-edc:
  tracing:
    serviceName: "my-fdsc-edc"

scorpio:
  tracing:
    serviceName: "scorpio"

keycloak:
  tracing:
    serviceName: "keycloak"
```

Components wired for tracing:

| Component | Mechanism | Default service name |
|---|---|---|
| IdentityHub | OTEL Java agent (init container) + `OTEL_*` env vars | `identityhub` |
| fdsc-edc | `OTEL_*` env vars via `values.yaml` | `fdsc-edc` |
| Keycloak | `OTEL_*` env vars via subchart `extraEnvVars` | `keycloak` |
| Scorpio | `QUARKUS_OTEL_*` env vars via subchart `extraEnvVars` | `scorpio` |
| tm-forum-api | `OTEL_*` env vars via subchart `additionalEnvVars` | `tm-forum-api` |
| contract-management | `OTEL_*` env vars via subchart `deployment.additionalEnvVars` | `contract-management` |
| marketplace | `OTEL_*` env vars via subchart (logic-proxy) `extraEnvVars` | `marketplace` |
| decentralizedIam | Forward-compatible passthrough (no runtime effect yet) | — |

### New values keys

The following top-level `values.yaml` keys are introduced:

| Key | Type | Default | Description |
|---|---|---|---|
| `tracing.enabled` | bool | `false` | Global on/off switch for OTEL tracing |
| `tracing.exporter.otlp.endpoint` | string | `""` | OTLP endpoint (defaults to bundled collector) |
| `tracing.exporter.otlp.protocol` | string | `"grpc"` | OTLP protocol (`grpc` or `http/protobuf`) |
| `tracing.exporter.otlp.insecure` | bool | `true` | Disable TLS on the OTLP exporter |
| `tracing.sampler` | string | `"parentbased_traceidratio"` | Trace sampler name |
| `tracing.samplerArg` | string | `"1.0"` | Sampler argument (probability for ratio samplers) |
| `tracing.resourceAttributes` | map | `{}` | Extra resource attributes for all workloads |
| `tracing.propagators` | string | `"tracecontext,baggage"` | Propagators to use |
| `opentelemetry-collector.enabled` | bool | `false` | Toggle the bundled OTEL Collector subchart |
| `identityhub.tracing.enabled` | bool | — | Per-component tracing override |
| `identityhub.tracing.serviceName` | string | `"identityhub"` | OTEL service name for IdentityHub |
| `identityhub.tracing.javaagent.enabled` | bool | `true` | Attach the OTEL Java agent via init container |

No existing values keys are modified or removed. Tracing is fully additive and opt-in.
